## 1. Goal & scope

**Goal:** A visitor can ask a question; we persist it and notify Praveen.

**In scope**
- A dedicated `/contact` page on the marketing site with a form: name, email,
  role (buyer / counselor / lender-realtor / other), message.
- Submission posts to a **public FastAPI endpoint** that writes a row to Supabase
  Postgres and sends an email to Praveen via Resend.
- Client + server validation, success/error UX, basic spam protection.

**Out of scope**
- Auto-reply / confirmation email to the submitter (requires a verified Resend
  domain we do not have yet). Notify-Praveen-only for now.
- Rewiring the product/audience CTAs that currently point to `#` — that is
  issue #11. This slice only wires the **Nav "Contact"** and **Footer "Contact"**
  links to `/contact`.
- Auth — the contact endpoint is intentionally public/unauthenticated.

**Done when:** submitting the form on the running app persists a row in Supabase
**and** delivers an email to Praveen; verified end-to-end.

## 2. Architecture & data flow

This follows PRD §7's data boundary: the browser never touches the database;
all data flows through FastAPI.

```
Visitor → /contact page (Next.js on Vercel)
   └─ submit → POST {NEXT_PUBLIC_API_URL}/contact   (public, no JWT)
        └─ FastAPI (Railway):
             1. validate payload (Pydantic)
             2. honeypot check (silently succeed if tripped)
             3. INSERT row → contact_messages (Supabase Postgres, async SQLAlchemy)
             4. send email → Resend REST API via httpx (notify Praveen) — non-fatal
             5. return 201
   └─ frontend shows success / error state
```

The contact endpoint is the project's **first unauthenticated write** and the
**first real app table** (the Alembic baseline has no app tables yet).

## 3. Backend (`api/`)

### New files
- **`app/models.py`** — introduces the SQLAlchemy 2.0 declarative `Base`
  (first ORM model in the project) and the `ContactMessage` model.
- **`app/schemas.py`** — Pydantic v2 request model `ContactCreate`.
- **`app/services/email.py`** — `async def send_contact_notification(...)` that
  POSTs to the Resend REST API (`POST https://api.resend.com/emails`,
  `Authorization: Bearer <key>`, JSON `{from, to, subject, html}`) using
  `httpx.AsyncClient`. **Why httpx, not the `resend` SDK:** the SDK is synchronous
  (`requests`-based) and would block the async event loop; `httpx` is already a
  project dependency, so the async email call adds **zero** new packages.
- **`app/routers/contact.py`** — `POST /contact`.

### Changed files
- **`migrations/env.py`** — set `target_metadata = Base.metadata` (currently
  `None`) and import models so they register, enabling `--autogenerate`.
- **`app/config.py`** — add settings: `resend_api_key`, `contact_notify_email`,
  `contact_from_email` (default `onboarding@resend.dev`), and
  `cors_origin_regex` (optional, for Vercel preview URLs — see §7).
- **`app/main.py`** — `app.include_router(contact.router)`; pass
  `allow_origin_regex=settings.cors_origin_regex` to the CORS middleware.
- **`pyproject.toml` / `uv.lock`** — only one new dep: `uv add "pydantic[email]"`
  (pulls in `email-validator` for `EmailStr`). No email SDK — email goes over the
  existing `httpx`.

### Data model — `contact_messages`
| column      | type          | notes                                   |
|-------------|---------------|-----------------------------------------|
| `id`        | uuid          | PK, server default `gen_random_uuid()` (Postgres 13+ core; Supabase has it) |
| `name`      | text          | not null, 1–100 chars                    |
| `email`     | text          | not null, validated as email             |
| `role`      | text          | not null; `CHECK` constraint limits it to the 4 role values |
| `message`   | text          | not null, 1–2000 chars                   |
| `created_at`| timestamptz   | not null, server default `now()`         |

Alembic owns this table (PRD §7). One autogenerated migration adds it; applied to
**both** Supabase dev and prod projects. The `role` CHECK constraint gives
DB-level integrity in addition to the Pydantic enum (cheap, matches the
zero-tech-debt goal).

### Request schema — `ContactCreate`
- `name`: str, 1–100
- `email`: `EmailStr`
- `role`: enum — `buyer` | `counselor` | `lender_realtor` | `other`
- `message`: str, 1–2000
- `website`: str, optional honeypot — **must be empty**; if filled, treat as spam

### Endpoint behavior — `POST /contact`
1. FastAPI validates the body against `ContactCreate` → invalid returns `422`.
2. Honeypot: if `website` is non-empty, **silently return `201`** without saving
   or emailing (defeats basic bots with zero user friction).
3. Persist the row (source of truth).
4. Send the Resend email. **Email failure is non-fatal:** log it and still return
   success — the durable record is the saved row.
5. Return `201` with a small JSON ack.

## 4. Frontend (`web/`)

### New files
- **`app/contact/page.tsx`** — the `/contact` route + page metadata; renders the
  form component. Server component shell.
- **`components/site/ContactForm.tsx`** — client component using the existing
  design tokens, `button-styles`, and shadcn primitives. Fields: name, email,
  role (select), message + a visually-hidden honeypot input. States:
  `idle → submitting → success | error`. Posts to
  `${NEXT_PUBLIC_API_URL}/contact`.

### Changed files
- **`components/site/Nav.tsx`** + **`components/site/Footer.tsx`** — point the
  "Contact" links to `/contact`.
- **`web/.env.local`** — `NEXT_PUBLIC_API_URL=http://localhost:8000` for local
  dev (Vercel already has it set, scoped per environment).

This is the first place `web/` actually consumes `NEXT_PUBLIC_API_URL`.

## 5. Error handling

- **Validation:** client-side for instant feedback **and** Pydantic server-side
  as the source of truth. A `422` maps to inline field errors.
- **Persist before email; email failure is non-fatal** — row saved = goal met.
- **Honeypot tripped** → silent success, no save, no email.
- **Network / 5xx** → friendly error state with a retry affordance; the form
  retains the user's typed values so nothing is lost.

## 6. Testing & verification

- **Backend (pytest)** — mirrors the existing no-network style in `conftest.py`.
  Note: `db.py` builds the `engine` at import from the real (lru-cached) settings,
  so tests must override the **`get_session` dependency** with a fake async session
  (overriding `get_settings` alone won't redirect the engine), and monkeypatch
  `send_contact_notification` so no email/network is hit:
  - validation errors return `422`,
  - happy path persists a row and calls the email sender,
  - honeypot path returns `201` and does **not** save or email,
  - email-failure path still returns `201`.
- **Frontend (Playwright MCP)** — load `/contact`, submit, see success; exercise
  validation and error states (CLAUDE.md rule 7: verify in a real browser).
- **End-to-end (the real "Done when")** — submit on the running app and confirm a
  row appears in Supabase **and** an email arrives in Praveen's inbox.
- **Local gates** — `api`: `ruff check . && ruff format --check . && mypy app &&
  pytest -q`; `web`: build + lint + typecheck (`/quick-test`).

## 7. Configuration & external setup

New environment variables (never committed):
- `RESEND_API_KEY` — Resend key (received; goes in `api/.env` + Railway dev & prod).
  ⚠️ Shared via chat, so treat as exposed; rotate before any real use.
- `CONTACT_NOTIFY_EMAIL` — notification recipient. **Must equal the email the
  Resend account was created with.** Verified Resend constraint: with the
  `onboarding@resend.dev` test sender (no verified domain), Resend returns **403
  "you can only send testing emails to your own email address"** for any other
  recipient. So this var = Praveen's Resend signup email, or sends fail.
- `CONTACT_FROM_EMAIL` — `onboarding@resend.dev` (Resend test sender; no domain yet).

Setup steps:
- Add the three vars to `api/.env`, `api/.env.example` (placeholders only), and
  both Railway services (dev + prod).
- Apply the new Alembic migration to both Supabase projects (dev + prod).

### ⚠️ CORS for preview deploys (verification finding)
The dev Railway service currently has `CORS_ORIGINS=https://keyzforme.vercel.app`
(the **production** alias, which tracks `main`). But the dev branch deploys to a
**preview** URL (`keyzforme-git-dev-…vercel.app`, plus per-commit hash URLs), and
`NEXT_PUBLIC_API_URL` on Preview points at the **dev** API. Result: a form
submitted from the dev preview would be **blocked by CORS** by the dev API.

Impact is limited — the two primary verification paths are unaffected:
- **Local:** `localhost:3000 → localhost:8000` (already in `CORS_ORIGINS`). ✅
- **Production (after promotion):** `keyzforme.vercel.app → prod API` (already
  allowed). ✅

To also test on the **dev preview**, do one of (recommended first):
1. Add `allow_origin_regex` to the FastAPI CORS middleware
   (e.g. `https://keyzforme-.*\.vercel\.app`) so all Vercel preview hashes are
   accepted — cleanest for Vercel's dynamic preview URLs. Needs a small
   `config.py` + `main.py` change (a new optional setting).
2. Or add the stable dev branch alias to the dev service's `CORS_ORIGINS`.

This slice will adopt option (1) so preview verification works without per-deploy
env edits.

## 8. Open decisions — resolved

1. **Flow:** browser → FastAPI → Supabase + Resend (not a direct-to-Supabase
   Server Action). The issue #3 text predates the FastAPI backend; PRD §7 wins.
2. **Spam:** honeypot field (no third-party captcha).
3. **Email failure:** non-fatal — submission still succeeds.
4. **CTA wiring:** only Nav/Footer "Contact" this slice; product CTAs deferred to #11.
5. **Email transport:** Resend REST API over the existing async `httpx`, not the
   blocking `resend` SDK (verified — avoids blocking the event loop + a new dep).
6. **Preview CORS:** add `allow_origin_regex` for `*.vercel.app` previews so the
   dev preview can call the dev API (verified gap, §7).

## 9. Known limitations (acceptable for the prototype)
- **No rate limiting** on the public endpoint — honeypot only. A determined actor
  could flood the table / inbox. Fine for a surprise prototype; revisit before
  real users (would pair with the auth/abuse work in later slices).
- **Resend test mode** can only deliver to the account email and only from
  `onboarding@resend.dev`. Sending to real visitors (auto-reply) needs a verified
  domain — out of scope here.
