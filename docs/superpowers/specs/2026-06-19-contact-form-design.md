## 1. Goal & scope

**Goal:** A visitor can ask a question; we persist it and notify Praveen.

**In scope**
- A dedicated `/contact` page on the marketing site with a form: name, email,
  role (buyer / counselor / lender-realtor / other), message.
- Submission posts to a **public FastAPI endpoint** that writes a row to Supabase
  Postgres and sends an email to Praveen via Resend.
- Client + server validation, success/error UX, basic spam protection.

**Out of scope**
- Auto-reply / confirmation email to the submitter (requires a verified Resend
  domain we do not have yet). Notify-Praveen-only for now.
- Rewiring the product/audience CTAs that currently point to `#` ‚Äî that is
  issue #11. This slice only wires the **Nav "Contact"** and **Footer "Contact"**
  links to `/contact`.
- Auth ‚Äî the contact endpoint is intentionally public/unauthenticated.

**Done when:** submitting the form on the running app persists a row in Supabase
**and** delivers an email to Praveen; verified end-to-end.

## 2. Architecture & data flow

This follows PRD ¬ß7's data boundary: the browser never touches the database;
all data flows through FastAPI.

```
Visitor ‚Üí /contact page (Next.js on Vercel)
   ‚îî‚îÄ submit ‚Üí POST {NEXT_PUBLIC_API_URL}/contact   (public, no JWT)
        ‚îî‚îÄ FastAPI (Railway):
             1. validate payload (Pydantic)
             2. honeypot check (silently succeed if tripped)
             3. INSERT row ‚Üí contact_messages (Supabase Postgres, async SQLAlchemy)
             4. send email ‚Üí Resend (notify Praveen) ‚Äî non-fatal on failure
             5. return 201
   ‚îî‚îÄ frontend shows success / error state
```

The contact endpoint is the project's **first unauthenticated write** and the
**first real app table** (the Alembic baseline has no app tables yet).

## 3. Backend (`api/`)

### New files
- **`app/models.py`** ‚Äî introduces the SQLAlchemy 2.0 declarative `Base`
  (first ORM model in the project) and the `ContactMessage` model.
- **`app/schemas.py`** ‚Äî Pydantic v2 request model `ContactCreate`.
- **`app/services/email.py`** ‚Äî thin Resend wrapper
  `send_contact_notification(...)`.
- **`app/routers/contact.py`** ‚Äî `POST /contact`.

### Changed files
- **`migrations/env.py`** ‚Äî set `target_metadata = Base.metadata` (currently
  `None`) and import models so they register, enabling `--autogenerate`.
- **`app/config.py`** ‚Äî add settings: `resend_api_key`, `contact_notify_email`,
  `contact_from_email` (default `onboarding@resend.dev`).
- **`app/main.py`** ‚Äî `app.include_router(contact.router)`.
- **`pyproject.toml` / `uv.lock`** ‚Äî `uv add resend` and `pydantic[email]`
  (for `EmailStr`).

### Data model ‚Äî `contact_messages`
| column      | type          | notes                                   |
|-------------|---------------|-----------------------------------------|
| `id`        | uuid          | primary key, server-generated           |
| `name`      | text          | not null, 1‚Äì100 chars                    |
| `email`     | text          | not null, validated as email             |
| `role`      | text          | not null, one of the 4 role values       |
| `message`   | text          | not null, 1‚Äì2000 chars                   |
| `created_at`| timestamptz   | not null, server default `now()`         |

Alembic owns this table (PRD ¬ß7). One autogenerated migration adds it; applied to
**both** Supabase dev and prod projects.

### Request schema ‚Äî `ContactCreate`
- `name`: str, 1‚Äì100
- `email`: `EmailStr`
- `role`: enum ‚Äî `buyer` | `counselor` | `lender_realtor` | `other`
- `message`: str, 1‚Äì2000
- `website`: str, optional honeypot ‚Äî **must be empty**; if filled, treat as spam

### Endpoint behavior ‚Äî `POST /contact`
1. FastAPI validates the body against `ContactCreate` ‚Üí invalid returns `422`.
2. Honeypot: if `website` is non-empty, **silently return `201`** without saving
   or emailing (defeats basic bots with zero user friction).
3. Persist the row (source of truth).
4. Send the Resend email. **Email failure is non-fatal:** log it and still return
   success ‚Äî the durable record is the saved row.
5. Return `201` with a small JSON ack.

## 5. Error handling

- **Validation:** client-side for instant feedback **and** Pydantic server-side
  as the source of truth. A `422` maps to inline field errors.
- **Persist before email; email failure is non-fatal** ‚Äî row saved = goal met.
- **Honeypot tripped** ‚Üí silent success, no save, no email.
- **Network / 5xx** ‚Üí friendly error state with a retry affordance; the form
  retains the user's typed values so nothing is lost.

## 6. Testing & verification

- **Backend (pytest)** ‚Äî mirrors the existing no-network style in `conftest.py`
  (DB session overridden, Resend mocked):
  - validation errors return `422`,
  - happy path persists a row and calls the email sender,
  - honeypot path returns `201` and does **not** save or email,
  - email-failure path still returns `201`.
- **Frontend (Playwright MCP)** ‚Äî load `/contact`, submit, see success; exercise
  validation and error states (CLAUDE.md rule 7: verify in a real browser).
- **End-to-end (the real "Done when")** ‚Äî submit on the running app and confirm a
  row appears in Supabase **and** an email arrives in Praveen's inbox.
- **Local gates** ‚Äî `api`: `ruff check . && ruff format --check . && mypy app &&
  pytest -q`; `web`: build + lint + typecheck (`/quick-test`).

## 8. Open decisions ‚Äî resolved

1. **Flow:** browser ‚Üí FastAPI ‚Üí Supabase + Resend (not a direct-to-Supabase
   Server Action). The issue #3 text predates the FastAPI backend; PRD ¬ß7 wins.
2. **Spam:** honeypot field (no third-party captcha).
3. **Email failure:** non-fatal ‚Äî submission still succeeds.
4. **CTA wiring:** only Nav/Footer "Contact" this slice; product CTAs deferred to #11.
